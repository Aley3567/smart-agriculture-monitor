# 新会话前端实现交接文档

## 一句话目标

把现有智慧农业平台前端改造成一个清晰区分 `板端实测 / 后端模型推算 / 室外气象接口 / 控制事件` 的成熟数据监控平台。主系统是我们自己的 Vue + FastAPI 平台，OneNET 只是旁路上报和云端可视化证明。

## 当前硬件与数据真相

当前板子真实能力：

| 指标 | 当前状态 | 来源口径 |
| --- | --- | --- |
| 温度 | 可真实采集 | DHT11，P2.0 |
| 空气湿度 | 可真实采集 | DHT11，P2.0 |
| 光照 | 板上有硬件，需 ADC 完成 | P0.7 光敏。ADC 未完成前只能标为板端模拟 |
| 土壤湿度 | 不能真实采集 | 后端模型推算 |
| CO2 | 不能真实采集 | 后端模型推算 |
| 土壤 EC | 不能真实采集 | 后端模型推算 |
| 土壤 TDS | 不能真实采集 | 后端模型推算 |
| 土壤肥力 | 不能真实采集 | 后端模型推算 |
| 红外状态 | 当前无 PIR 模块 | 后端模型推算，按 PIR 人体/活动事件处理 |

不要把所有数据都叫“实测”。UI 必须显示低调但明确的来源标签。

## 最终侧边栏

侧边栏只保留 6 项：

1. 实时监测
2. 历史分析
3. 控制与阈值
4. 模型管理
5. 环境气象
6. 告警日志

合并规则：

- `设备管理` 不单独占侧边栏，合并到 `控制与阈值` 页的 `设备映射` 区。
- `系统设置` 不单独占侧边栏，放到右上角 `管理员` 下拉菜单。
- 原 `历史数据` 改名为 `历史分析`。
- 原 `阈值配置` 改名为 `控制与阈值`。
- 保留 `环境气象`，但定位为室外参考。

## 顶部状态栏

顶部建议显示：

```text
主平台运行中 · OneNET旁路上报正常 · 串口网关已连接
```

含义：

- 主平台：我们自己的前后端系统。
- OneNET：旁路上报，不是主控制链路。
- 串口网关：`serial_bridge` 到 FastAPI 的连接状态。

## 数据来源标签

统一来源标签：

| 内部值 | 页面标签 | 用途 |
| --- | --- | --- |
| `measured` | 实测 | 板端真实传感器 |
| `simulated_backend` | 模型 | 后端模型推算 |
| `simulated_firmware` | 板端模拟 | 固件临时模拟值 |
| `api` | 气象接口 | Open-Meteo/高德等外部数据 |
| `control` | 控制 | 自动控制或手动控制事件 |
| `system` | 系统 | 串口、OneNET、服务状态 |

推荐卡片标签示例：

```text
温度 26.4°C         实测 · DHT11 P2.0
空气湿度 58%        实测 · DHT11 P2.0
光照 820 lux        实测 · P0.7光敏
土壤湿度 43.5%      模型
CO2 612 ppm         模型
室外温度 27.6°C     气象接口 · Open-Meteo
```

如果光照 ADC 还没完成，光照标签应临时为：

```text
板端模拟
```

不要写 `BH1750`，当前硬件不是 BH1750。

## Grilling 设计决策结论

以下是已确认的关键设计决策，新会话不要重新推翻，除非发现新的硬件证据。

### 1. 土壤湿度口径

土壤湿度正式定义为算法仿真值，不称为板端实测值。

推荐说法：

```text
土壤湿度由真实温度、真实空气湿度、真实光照、水泵状态和时间步长驱动的水量平衡模型生成。
```

### 2. 天气 API 与土壤模型关系

天气 API 不作为土壤湿度主驱动，只做参考展示和可选初始化辅助。

原因：室外降水、室外土壤含水量不等于桌面/棚内实验环境。

### 3. 仿真模型放置位置

仿真放后端，不放固件，不放前端。

数据流：

```text
CC2530真实采集 temp/humi/light
  -> serial_bridge
  -> FastAPI 后端
  -> 后端模型补齐 soil/co2/ec/tds/fertility/infrared
  -> 数据库 / WebSocket / 前端 / OneNET旁路
```

### 4. 串口协议中的 `s`

短期为了兼容可保留 `s` 字段，但后端不信任它；最终协议应只传真实字段 `t/h/l`。

短期：

```text
t:25-h:60-l:820-s:45
```

后端忽略 `s`，重新生成 `soil`。

长期：

```text
t:25-h:60-l:820
```

### 5. 光照处理

光照短期使用固件传来的 `light`，但标记为 `板端模拟`；补完 P0.7 ADC 后标记为 `实测 · P0.7光敏`。

不要使用室外天气 API 辐射替代棚内光照。

### 6. CO2 模型

CO2 依赖：

- 光照
- 温度
- 红外事件
- 时间段
- 固定自然通风系数

CO2 不绑定第三路执行器，因为第三路按任务书应是 `灭虫灯`，不是天窗/通风。

### 7. TDS / EC / 土壤肥力

内部维护 `soil_ec` 主状态，再推导：

```text
soil_tds ≈ soil_ec * 640
soil_fertility = normalize(soil_ec)
```

施肥泵、水泵、光照、温度、土壤湿度共同影响 EC。

### 8. 红外

红外按 PIR 人体/活动检测模拟，字段为 `infrared`，取值 0/1。

不要把红外等同于灭虫灯。红外是传感器状态，灭虫灯是执行器。

### 9. 自动控制

自动控制允许使用模型字段，但必须保留来源标记。

| 控制 | 依据 | 口径 |
| --- | --- | --- |
| 水泵 | 土壤湿度 · 模型 | 自动控制可启用 |
| 施肥泵 | 土壤肥力/EC · 模型 | 手动优先，给建议 |
| 灭虫灯 | 手动控制；红外只告警 | 不自动等同红外 |
| 流水灯报警 | 实测/模型都可触发 | 告警日志保留来源 |

### 10. 前端来源标签

前端必须显示 `实测 / 模型 / 气象接口 / 控制 / 系统` 标签，但要低调展示，不能把页面做成“全是假的”。

### 11. 数据库存来源

建议 `sensor_data` 增加 `sources` JSON/TEXT 字段，记录每个字段来源。

示例：

```json
{
  "temp": "measured",
  "humi": "measured",
  "light": "measured",
  "soil": "simulated_backend",
  "co2": "simulated_backend",
  "soil_ec": "simulated_backend",
  "soil_tds": "simulated_backend",
  "soil_fertility": "simulated_backend",
  "infrared": "simulated_backend"
}
```

### 12. 模型状态持久化

模型状态运行时存在后端内存；每次采样结果入库；后端重启时从最新一条 `sensor_data` 恢复状态。

不需要先做复杂的单独 `simulation_state` 表。

### 13. OneNET 定位

OneNET 纳入正式方案，但只作为旁路上报和可视化证明，不作为主控制链路。

推荐架构：

```text
主链路:
CC2530 -> 串口网关 -> FastAPI -> Vue前端

旁路:
FastAPI -> OneNET物模型 -> OneNET大屏/控制台截图
```

### 14. Dashboard 展示

Dashboard 必须分区：

- 板端实测
- 模型推算
- 趋势分组视图
- 执行器控制
- 最近告警
- 室外气象参考

不要把所有指标并排展示成同等级实测传感器。

## 页面规格

### 实时监测

目标：作为主屏，快速说明系统状态和数据来源。

必须包含：

- 板端实测：温度、空气湿度、光照。
- 模型推算：土壤湿度、CO2、土壤 EC、土壤 TDS、土壤肥力、红外状态。
- 趋势分组视图：实测趋势、土壤模型趋势、CO2/红外事件。
- 执行器控制：水泵、施肥泵、灭虫灯。
- 最近告警：每条告警显示来源。
- 室外气象参考：明确不作为棚内实测来源。

### 历史分析

目标：查看历史趋势、来源、导出数据。

必须包含：

- 时间范围筛选。
- 粒度选择。
- 实测趋势图。
- 模型趋势图。
- 数据质量面板。
- 数据表格，保留来源列。

注意：

- 模型字段数量应为 6 个：土壤湿度、CO2、EC、TDS、土壤肥力、红外。

### 控制与阈值

目标：配置阈值、控制策略、设备映射。

必须包含：

- 实测阈值：温度、空气湿度、光照。
- 模型阈值：土壤湿度、CO2、土壤 EC、土壤肥力、红外触发策略。
- 执行器策略：水泵、施肥泵、灭虫灯。
- 设备映射：P0.6、P1.0、P1.1、P1.6。

命令口径：

- 水泵：`BLEGLED1` / `BLEKLED1`
- 施肥泵：`BLEGLED2` / `BLEKLED2`
- 灭虫灯：`BLEGLED3` / `BLEKLED3`

### 模型管理

目标：解释和管理后端模型，证明不是随机数。

必须包含：

- 土壤水量平衡模型。
- CO2 质量守恒模型。
- EC/TDS/肥力模型。
- 红外事件模型。
- 来源映射表。
- 模型输出预览。

注意：

- 红外当前是模型，不是实测。

### 环境气象

目标：室外气象参考，不替代板端传感器。

必须包含：

- 位置选点。
- 室外温度、湿度、降水、风速、风向、辐射、AQI。
- 室外参考 vs 棚内数据对比。
- 使用边界说明：
  - 不替代板端传感器。
  - 不直接控制水泵。
  - 可用于报告对比和初始参考。
  - 断网不影响主平台控制。

### 告警日志

目标：告警追溯和控制动作解释。

必须包含：

- 筛选：时间、级别、来源、状态。
- KPI：今日告警、未确认、模型告警、实测告警、自动控制。
- 告警表格。
- 告警详情。
- 决策追溯：输入数据 -> 阈值规则 -> 动作指令 -> 当前状态。
- 处理记录。

每条告警必须显示来源：实测、模型、控制、系统、气象接口。

第 3 课：数据怎样上行

“上行”就是数据从硬件现场一路走到后端和前端。本课只追一条数据：t:25-h:60-l:800-s:45。

1 角色 2 硬件 3 上行 4 下行 5 实验 术语表

先看原始数据

终端节点发出的不是 JSON，也不是数据库行，而是一行简单字符串：

t:25-h:60-l:800-s:45

t
temp 温度
h
humi 湿度
l
light 光照
s
soil 土壤

一步步走完整条上行链路

1

终端节点准备数据

终端 CC2530 读取 DHT11，再生成光照/土壤模拟值，拼成 t/h/l/s 字符串。

2

ZigBee 发给协调器

终端通过 AF_DataRequest 把数据发给协调器。这里是 CC2530 之间的无线。

3

协调器转 UART

协调器收到无线数据后，不解析业务，只原样写到 UART0，并补上换行。

4

bridge 解析串口

parse_serial_line() 把 t/h/l/s 映射成 temp/humi/light/soil。

5

WebSocket 发给后端

bridge 封装成 type: sensor_data 的 JSON，发到后端 /ws/bridge。

6

后端入库并广播

FastAPI 写入 SensorData，同时通过 /ws/data 推给前端。

新手要理解的关键转换

这条链路里最关键的是“格式转换”：

串口字符串 → Python dict → WebSocket JSON → 数据库记录 → 前端展示数据

所以 bridge 的价值不是“多余的一层”，而是把硬件串口世界接到后端网络世界。
